# 記憶醸造所（Memory Brewery）プロダクト仕様書

- 文書バージョン: 0.1
- ステータス: Draft
- 想定用途: ハッカソン向けMVP / 仕様駆動開発
- 対象地域: 石川県
- 更新日: 2026-07-18

## 1. 概要

### 1.1 プロダクト名
**記憶醸造所（Memory Brewery）**

### 1.2 一文で表すと
石川の日本酒が持つ個性を土台に、石川の土地と飲み手の記憶を三段仕込みし、音楽と動くビジュアルとして開栓する参加型Web体験。

### 1.3 キャッチコピー
**酒を評価するのではなく、酒が生んだ記憶を醸造する。**

補助コピー:
- 一人一杯、みんなで一仕込み。
- 米から酒を。記憶から音と光を。

## 2. 背景と課題

日本酒の価値は、銘柄・製法・成分・甘辛などの属性だけでなく、「誰と、どこで、どんな気持ちで飲んだか」という体験にも宿る。一方、その体験は言葉にしにくく、その場限りで消費されやすい。

既存の記録手段は、商品スペック、星評価、写真、文章投稿を中心としており、味わった瞬間の空気や感情を、別の人と共有・再体験する仕組みは弱い。

また、日本酒関連の統計データは存在する一方、アプリで利用しやすい銘柄・商品単位の公式データは限られる。そこで本プロダクトでは、少数銘柄の日本酒データを整理し、石川のオープンデータおよび参加者の入力と組み合わせる。

## 3. 解決方針

生成作品を次の3層で構成する。

1. **日本酒の個性**: 作品の基礎となる音・形・動き
2. **石川の土地**: 作品の背景・空気・地域性
3. **人の記憶**: 作品を個人化し、物語を生む要素

参加者は三段仕込みを模した3回の操作を行う。

- 初添: いま感じたもの
- 仲添: 石川の土地の記憶
- 留添: 未来に開けたい場面

三段仕込みの完了後、スマートフォン上で個人作品を「開栓」する。同時に、個人作品の要素が会場全体の集合作品へ合流する。

## 4. 対象ユーザー

### 4.1 主対象
- 日本酒イベント、試飲会、地域イベントの参加者
- 日本酒初心者
- 石川県の文化・観光に関心を持つ人
- 酒を飲めないが、香り・ラベル・土地の物語から参加したい人

### 4.2 運営者
- ハッカソンのデモ運営者
- 酒蔵、酒販店、観光・地域イベントの主催者

## 5. ユーザー体験

### 5.1 体験時間
1人あたり約90〜120秒を目標とする。

### 5.2 体験フロー

#### Step 0: 一献
QRコードから対象の日本酒の体験ページを開く。

表示内容:
- 酒蔵名
- 商品名
- 最小限の商品紹介
- 「この一杯から生まれた記憶を、三回に分けて仕込みます。」

参加者が飲酒できない場合は、香り、ラベル、商品説明、土地の情報から参加できる。

#### Step 1: 初添 — いま感じたもの
問い:
**「この一杯を、色と動きで表すなら？」**

操作:
- 基調色を選ぶ
- 画面上を指でなぞり、線・波・渦などを描く
- 指の速さや長さを取得する

反映例:
- 色 → 基調色
- 線の形 → ビジュアルモチーフ
- 指の速度 → 音楽テンポまたは動きの速度
- タッチ時間 → 音の余韻

操作後、数秒の個人プレビューを再生する。

#### Interlude: 踊り
初添の直後に10秒程度の待機演出を入れる。

表示:
- 「仕込みを休ませています。」
- 「もう一度、香りや余韻を感じてください。」

踊りは入力工程として数えず、初添と仲添の間の演出とする。

#### Step 2: 仲添 — 石川の土地の記憶
問い:
**「この一杯に、どの石川を重ねますか？」**

対象酒の酒蔵所在地・酒米産地などを基に、石川のオープンデータから生成した候補を3件程度表示する。

候補例:
- 日本海の冬の風
- 金沢の雪明かり
- 白山から流れる水
- 能登の祭りの火
- 加賀の工芸の色

候補は単なる観光コピーではなく、元データと出典を保持する。

反映例:
- 気温 → 色温度・音域
- 降水・降雪 → 粒子密度・細かな打音
- 風向・風速 → ビジュアルの流れ
- 日照 → 明るさ・和音の開放感
- 場所・文化 → 模様・音色・空間表現

#### Step 3: 留添 — 未来の一杯
問い:
**「この酒を、次はどんな場面で開けたいですか？」**

入力方法:
- 事前定義した候補
- 20〜40文字程度の短い自由入力

候補例:
- 次の雪の日
- 友人との再会
- 家族の祝い
- 一人で静かに過ごす夜
- 石川をもう一度訪れたとき

反映例:
- 再会 → 線が合流する
- 一人 → 中央へ収束する
- 旅 → 外側へ広がる
- 祝い → 光・リズムが増える

#### Step 4: 開栓
表示:
- 「三段の仕込みが完了しました。」
- 「あなたの記憶酒を開栓します。」

15〜30秒の個人作品をスマートフォンで再生する。

表示内容:
- 自動生成された記憶酒名
- 音楽
- 動くビジュアル
- 初添・仲添・留添の内容
- 会場全体の作品へのリンクまたは参加状態

## 6. 会場全体の体験

会場スクリーンには、参加者全員の入力から成長する集合作品を表示する。

### 6.1 集合作品の原則
- 新しい参加者が加わるたびに変化する
- 単純に音数を増やし続けず、役割を割り当てる
- 個人作品と集合作品は同一にしない
- 個人作品の特徴を、集合作品の一部として識別できる余地を残す

### 6.2 変化の例
- 新しい色・形が加わる
- 土地レイヤーの割合が変化する
- 音の層や音色が変化する
- 記憶同士が線や波で接続される
- 参加人数に応じて作品が発展する

### 6.3 会場作品の開栓
一定人数または一定時間で、30〜60秒の完成版を再生する。

## 7. データ仕様

### 7.1 日本酒データ
MVPでは石川県の日本酒1〜3商品を対象とし、手動で検証済みデータを作成する。

必須項目:
- sake_id
- brewery_name
- product_name
- brewery_address
- brewery_latitude
- brewery_longitude
- category / brewing_method
- rice_variety
- rice_origin（取得可能な場合）
- polishing_ratio
- alcohol_percentage
- official_description
- source_url
- usage_rights_note

任意項目:
- sake_meter_value
- acidity
- amino_acid_value
- yeast
- brewing_water
- pasteurization
- aging_period
- recommended_temperature
- food_pairing
- product_image_url

注意:
- 商品説明・画像の無断再配布は行わない
- 公開時は出典と利用条件を記録する
- 数値と味の因果関係を科学的事実として断定しない

### 7.2 石川オープンデータ
候補:
- 観光施設一覧
- 観光イベント一覧
- 文化財一覧
- 気象データ
- 地理・位置情報
- 必要に応じて地産地消・施設データ

最低限保持する属性:
- source_dataset_id
- source_title
- source_url
- license
- record_id
- name
- description / summary
- category
- municipality
- latitude
- longitude
- date / season
- tags
- raw_values
- normalized_generation_parameters

採用前に確認する項目:
- 行数
- 欠損率
- 更新日
- ライセンス
- 緯度経度の充足率
- 説明文・画像の充足率
- 市町別の表記差
- APIまたはファイル取得方法

### 7.3 参加者入力
- session_id
- participant_id（匿名・一時ID）
- sake_id
- hatsuzoe_color
- hatsuzoe_gesture_points
- hatsuzoe_gesture_speed
- naka_choice_id
- tome_choice / free_text
- generated_recipe
- created_at
- consent_flags

個人を直接特定する情報は収集しない。

## 8. 生成仕様

### 8.1 生成の考え方
生成AIは完成音源・完成映像を直接生成せず、入力を構造化した「醸造レシピ」へ変換する。

音楽・映像はブラウザ上で決定論的または半決定論的に生成する。

### 8.2 醸造レシピ例
```json
{
  "tempo": 68,
  "scale": "minor_pentatonic",
  "instrument_set": ["glass", "soft_pad", "water"],
  "brightness": 0.34,
  "warmth": 0.45,
  "motion_type": "slow_ripple",
  "motion_speed": 0.42,
  "particle_density": 0.28,
  "reverb": 0.72,
  "ending_type": "converge",
  "memory_title": "雪間の再会"
}
```

### 8.3 三層の合成
- 酒レイヤー: 基礎テンポ、音色、形状、密度
- 土地レイヤー: 背景、流れ、環境音、色温度
- 記憶レイヤー: 色、ジェスチャー、展開、終わり方

### 8.4 変換ルールの透明性
- 変換ルールは作品上の演出規則である
- 味覚や製法を科学的に音へ変換したと誤認させない
- 「この入力がどの要素へ反映されたか」を結果画面で説明可能にする

## 9. 機能要件

### FR-01 対象酒の読み込み
QRコードまたはURLからsake_idを指定し、対象酒のデータを読み込める。

### FR-02 初添入力
色選択とタッチジェスチャーを取得できる。

### FR-03 踊り演出
初添と仲添の間に、短い自動演出を表示できる。

### FR-04 仲添候補生成
対象酒と地域データを基に、土地の記憶候補を3件程度提示できる。

### FR-05 留添入力
未来の場面を選択または短文で入力できる。

### FR-06 醸造レシピ生成
三段の入力と基礎データから、検証済みスキーマのJSONを生成できる。

### FR-07 個人作品再生
スマートフォン上で、音楽と動くビジュアルを15〜30秒再生できる。

### FR-08 集合作品への反映
個人作品の要素を、会場作品へ数秒以内に反映できる。

### FR-09 記憶酒表示
記憶酒名と三段の入力内容を表示できる。

### FR-10 フォールバック
生成AIや通信が失敗しても、ルールベースの醸造レシピで作品を完成できる。

## 10. 非機能要件

### NFR-01 モバイル対応
主要なスマートフォンブラウザで操作可能であること。

### NFR-02 操作時間
三段入力完了まで90秒以内を目標とする。

### NFR-03 応答性能
各画面遷移は原則2秒以内。AI解析は10秒以内を目標とし、待機中は醸造演出を表示する。

### NFR-04 音声再生
ブラウザの自動再生制限を考慮し、ユーザー操作をトリガーに音声を開始する。

### NFR-05 アクセシビリティ
- 音を出せない環境でもビジュアルで体験できる
- 色だけに意味を依存しない
- 飲酒を必須にしない
- 文字サイズとコントラストを確保する

### NFR-06 プライバシー
個人を直接特定する情報を収集せず、入力内容の公開範囲を明示する。

### NFR-07 安定性
ネットワーク不調時でも、個人端末内のルールベース生成で最低限の体験を完了できる。

## 11. MVP範囲

### 必須
- 対象酒1〜3商品
- QRコードから体験開始
- 初添・踊り・仲添・留添
- AIまたはルールベースによる醸造レシピ生成
- Tone.js等による音楽生成
- Canvas / p5.js等による動くビジュアル
- スマートフォン上で個人作品を再生
- 複数人入力を会場作品へ反映
- フォールバック動作

### 余裕があれば
- 参加後の再生URL
- 個人作品の共有
- 複数銘柄比較
- 会場作品の保存
- 過去イベント作品の再生
- 多言語対応

### 非対象
- 全国の銘柄検索
- EC・購入導線
- SNS機能
- 高度なレコメンド
- 恒久的ユーザーアカウント
- 完全なAI音楽・動画生成
- Tableau連携

## 12. 受け入れ基準

### AC-01 三段仕込み完了
参加者が、初添・仲添・留添の3入力を行い、開栓画面へ到達できる。

### AC-02 酒による差
同じ参加者入力でも、対象酒が異なると音またはビジュアルの基礎表現が変わる。

### AC-03 土地による差
仲添で異なる土地候補を選ぶと、作品の背景・音色・動きのうち少なくとも1要素が変わる。

### AC-04 人による差
初添の色・ジェスチャーまたは留添の入力が異なると、個人作品が変わる。

### AC-05 モバイル完結
参加者は会場スクリーンを見なくても、スマートフォンだけで個人作品を再生できる。

### AC-06 集合作品
2人以上の参加で、会場作品に複数参加者の要素が反映される。

### AC-07 失敗時継続
AI APIが失敗しても、参加者はエラー画面で停止せず、作品を開栓できる。

### AC-08 出典管理
使用した日本酒情報とオープンデータについて、出典URLと利用条件を確認できる。

## 13. 技術構成案

- フロントエンド: React / Next.js
- 音楽: Tone.js
- ビジュアル: Canvas API / p5.js
- バックエンド: API Gateway + Lambda または同等構成
- データストア: DynamoDB / S3 または軽量DB
- 生成AI: Amazon Bedrock
- リアルタイム反映: WebSocket。MVPでは短間隔ポーリングでも可
- デプロイ: AWSまたは静的ホスティング＋API

技術選定は実装フェーズでADR（Architecture Decision Record）として確定する。

## 14. テスト方針

### 単体テスト
- 入力値から醸造レシピへの変換
- 数値正規化
- フォールバックルール
- JSONスキーマ検証
- 個人データを含まないことの確認

### 結合テスト
- 三段仕込みの画面遷移
- AI API成功・失敗
- データ取得
- 個人作品生成
- 集合作品への反映

### E2Eテスト
- モバイル画面幅でQR流入から開栓まで
- 音声許可・拒否の両方
- 通信遅延・API失敗
- 2端末以上の参加

### 手動体験テスト
- 90〜120秒で完了できるか
- 日本酒らしさが説明なしでも伝わるか
- 自分の入力が作品に反映されたと感じられるか
- 大画面がなくても個人体験として成立するか

## 16. 開発の進め方（仕様駆動）

1. 本仕様書をレビューし、MVP境界と未決事項を確定する
2. ユーザーストーリーと受け入れ基準をIssue化する
3. データ調査・利用許諾を実装と分離して先行する
4. 画面遷移と生成ルールをプロトタイプで検証する
5. ADRで技術判断を記録する
6. 各Issueは受け入れ基準とテストを満たしてから完了とする
7. ハッカソン前に実機・複数端末・不安定回線でリハーサルする
