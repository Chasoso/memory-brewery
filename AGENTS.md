# AGENTS.md

このリポジトリでは、仕様書と承認済みプロトタイプを正本として、Codex を含む開発者が Issue 単位で安全に作業します。

## 正本と変更順序

- プロダクト挙動の正本は `docs/product/memory_brewery_product_spec_v0.1.md`。
- 視覚・操作設計の正本は `prototypes/high-fidelity-v0.1/`。
- 仕様変更は実装前に仕様書へ、デザイン変更は実装前にプロトタイプまたは `DESIGN.md` へ反映する。
- 実装都合で色、タイポグラフィ、余白、画面構成、三段仕込みの順序、主要アニメーションを変更しない。

## 作業単位と Git

- `main` へ直接 push しない。`develop` ブランチは作らない。
- 原則として一つの Issue を一つのブランチ・PR で扱い、無関係な変更を混ぜない。
- ブランチ名は `<type>/issue-<number>-<short-name>` とする。例: `feat/issue-42-brewing-recipe`、`docs/issue-43-data-catalog`。
- 初期基盤のように対応 Issue がまだない作業は、所有者の明示的な依頼時だけ `chore/bootstrap-<short-name>` を使える。以後の変更は Issue 番号を必須とする。
- Conventional Commits を使用し、`--no-verify` を使わない。
- PR 作成前に差分、受け入れ基準、正本との整合、秘密情報を自己レビューする。
- PR は対象 Issue、変更内容、検証結果、受け入れ基準を記載し、完了した Issue にだけ `Closes #N` を含める。Codex は PR を自己判断でマージしない。

## 実装の境界

- この段階でフレームワーク、クラウド、デプロイ方式を決めない。デプロイ設定やクラウド資源を追加しない。
- 主要機能は外部 API がなくても fixture、mock、ルールベースのフォールバックで検証可能にする。
- 参加者画面と会場画面はローカルの複数タブまたは複数ブラウザで検証できるようにする。
- 本番コードは TypeScript と strict 型チェックを使用する。外部入力と fixture はスキーマ検証し、醸造レシピは明示的な型で表す。
- ジェネラティブ表現には seed を注入可能にし、テストでは乱数、時刻、アニメーションを固定可能にする。
- 音声が使えなくても操作を完了できるようにし、UI、ドメインロジック、音楽生成、ビジュアル生成を分離する。

## 検証と停止条件

- [検証方針](docs/development/validation-policy.md) に従い、実行したコマンド、結果、未実行の理由を PR に記録する。
- 既定の検証はネットワークを使わない。外部連携は明示的に opt-in とする。
- 失敗が変更範囲内なら修正して再実行する。範囲外、原因不明、または正本と矛盾する場合は止めて報告する。
- package / lockfile、CI、品質ゲート、正本、プロトタイプへの変更は、Issue の目的と必要性を PR で明記する。
